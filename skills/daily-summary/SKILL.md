# Daily Summary スキル

## Vaultのフォルダ構造

```
Vault/
 ├ 00_Inbox/          ← Web Clipperの吐き出し先（常にフラット）
 ├ 01_Clips/          ← 整理済みアーカイブ（YYYYMMDD/単位）
 ├ 02_DailySummary/   ← 日次まとめ（1ファイル/日）
 └ 99_Logs/           ← 処理ログ（1ファイル/日）
```

## フォルダの状態定義

| フォルダ | ファイルがある意味 |
|---|---|
| 00_Inbox | 未処理（要対応） |
| 01_Clips | 整理済み・アーカイブ済み |
| 02_DailySummary | まとめ生成済み |
| 99_Logs | 処理記録 |

## 基本原則

- **まとめ生成を先に、フォルダ整理を後に行う**（整理が失敗してもまとめは残す）
- 処理はClaudeが行う（外部APIやネットワークアクセスは使用しない。元記事URLへの再アクセスもしない。入力はMD本文のみを信頼する）
- MDファイルのみを対象とする
- **冪等性を保つ**：再実行しても二重処理・二重登録が起きないようにする

---

## 処理状態の管理

フォルダ位置だけでなく、各クリップMDのfrontmatterに処理状態を記録する。

### 処理済みマーク

処理完了時、各クリップMDのfrontmatterに以下を追記する：

```yaml
processed: true
processed_at: "YYYY-MM-DDTHH:MM"
summary_date: "YYYYMMDD"
```

### 判定ルール

- `processed: true` が既にあるファイルは処理対象から除外する
- 01_Clips 内に同一ファイル名が存在する場合も除外する

---

## 重複判定

同じ記事の二重処理を防ぐため、以下の順で重複をチェックする：

1. **source URL 完全一致**：frontmatterの `source:` が同日の処理済み記事と一致 → スキップ
2. **title + date 一致**：タイトルと日付が完全一致 → スキップ
3. 重複と判定した場合、ログに「重複スキップ」として記録し、ファイルは 00_Inbox に残す

---

## Web Clipperが生成するfrontmatter形式

```yaml
---
title: 記事タイトル
source: https://example.com
date: YYYY-MM-DD
tags: []
---
```

### frontmatter補完ルール（欠損時の優先順位）

各フィールドが欠損している場合、以下の優先順位で補完する：

| フィールド | 優先順 |
|---|---|
| title | frontmatter.title → 本文の最初のH1（`# `） → ファイル名（拡張子除く） → 本文冒頭1文（50字まで） |
| source | frontmatter.source → 本文中の最初のURL → 空欄（`""`） |
| date | frontmatter.date → ファイル名に含まれる日付パターン → 処理実行日 |
| tags | frontmatter.tags → `[]` で初期化 |

frontmatter自体が存在しない場合は、上記ルールで全フィールドを生成し、ファイル先頭に `---` ブロックを追加する。

## タグ体系（DailySummary生成時に各記事へ付与すること）

各クリップMDファイルの `tags:` に、以下のタグ体系から適切なものを選んで書き込む。
Obsidianの階層タグ形式（`ジャンル/サブカテゴリ`）を使用する。

### ジャンル（何の分野か・複数可）

| タグ | 用途 | 判定基準 |
|---|---|---|
| `ジャンル/AI` | AI・機械学習・LLM全般 | 記事の主題がAI技術・モデル・AI活用の場合 |
| `ジャンル/テック` | IT技術・ソフトウェア・ハードウェア | プログラミング・OS・デバイス等が主題の場合 |
| `ジャンル/ビジネス` | 経営・マーケティング・業界動向 | 企業戦略・市場分析・業界再編等が主題の場合 |
| `ジャンル/セキュリティ` | 情報セキュリティ・サイバー攻撃 | 脆弱性・攻撃手法・セキュリティ対策が主題の場合 |
| `ジャンル/ネットワーク` | インフラ・通信・クラウド | NW機器・回線・クラウドサービス基盤が主題の場合 |
| `ジャンル/ツール` | アプリ・サービス・プロダクト紹介 | 特定ツール・サービスの紹介やレビューが主題の場合 |

**迷いやすいケースの判断基準：**
- AIツールの紹介記事 → `ジャンル/AI` と `ジャンル/ツール` の両方を付与
- セキュリティ事故の経営影響 → `ジャンル/セキュリティ` のみ（ビジネスは事故の「主題」ではない）
- クラウドサービスのセキュリティ機能 → `ジャンル/ネットワーク` と `ジャンル/セキュリティ` の両方

### 重要度（1つだけ）

| タグ | 用途 | 判定基準 |
|---|---|---|
| `重要度/高` | 業務への影響大 | **今週中に**対応判断・調査・共有が必要なもの。興徳グループの業務や担当システムに直接影響するもの |
| `重要度/中` | 近いうちに活用可能 | **今月中に**理解・活用の価値があるもの。スキルアップや業務改善に使えそうなもの |
| `重要度/低` | トレンド把握 | 直接の業務関連はないが、知っておくと視野が広がるもの。迷ったらこれ |

### ソース（情報元の種類・1つだけ）

| タグ | 用途 | 判定基準 |
|---|---|---|
| `ソース/公式` | 公式ブログ・リリースノート・ドキュメント | ベンダー・サービス運営元の一次発信（Microsoft Blog、AWS News、Anthropic Blog等） |
| `ソース/ブログ` | 個人ブログ・Qiita・Zenn・note | 個人や組織の技術ブログ・解説記事 |
| `ソース/ニュース` | ニュースメディア | ITmedia・TechCrunch・Gigazine・NHK等のメディア記事。企業寄稿や転載も掲載メディアで判断 |
| `ソース/SNS` | Reddit・X・Hacker News等 | ソーシャルメディア上の投稿・スレッド |

### タグ付けのルール

- ジャンルは複数付けてよい（例：AIとツール両方該当なら両方付与）
- 重要度・ソースは必ず1つだけ選ぶ
- 迷ったら `重要度/低` を選ぶ
- `重要度/高` は業務直結の場合のみにとどめる

### タグ付けの例

```yaml
tags:
  - ジャンル/AI
  - ジャンル/ツール
  - 重要度/中
  - ソース/SNS
```
