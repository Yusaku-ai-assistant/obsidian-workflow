# Daily Summary コマンド

## 実行前チェック

1. `00_Inbox` フォルダを確認する
2. MDファイルが1件以上あるか確認する
3. **ない場合**：「00_Inboxにファイルがありません。Web Clipperで記事を保存してから実行してください。」と伝えて終了する

---

## 処理フロー概要

```
1. 00_Inbox の md を列挙 → 今回の処理対象を確定（以降この一覧は変更しない）
2. 既処理・重複判定（processed: true のファイルを除外、URL正規化で重複検出）
3. frontmatter補完（欠損フィールドを優先順位に従い補完）
4. タグ付与（ジャンル・重要度・ソース。既存タグは保持し追加のみ行う）
5. 記事要約生成 → 02_DailySummary/YYYYMMDD.md に書き込み
6. ↑ 成功した記事にのみ processed: true を付与（途中失敗の取りこぼし防止）
7. 99_Logs/YYYYMMDD.md にログ記録
8. 01_Clips/YYYYMMDD/ に処理成功ファイルを移動（ファイル名衝突時はリネーム）
9. 失敗したファイルは 00_Inbox に残し、failure_reason をfrontmatterに記録
```

---

## 処理手順（必ずこの順番で実行すること）

### Step 1：Inboxの全MDファイルを読み込む

- `00_Inbox` 内の全MDファイルを読み込む
- 各ファイルからfrontmatterと本文を取得する
- **この時点で処理対象リストを確定する。以降のステップで00_Inboxへの新規追加があっても今回の処理対象には含めない**

### Step 1.1：既処理・重複判定

- `processed: true` がfrontmatterにあるファイルは **スキップ**（ログに「既処理スキップ」として記録）
- 同日の処理済み記事と `source:` URLを**正規化した上で**比較し、一致するファイルは **スキップ**（ログに「重複スキップ」として記録）
- `title:` と `date:` が共に一致するファイルも **スキップ**
- `failure_reason` が付いているファイルは再処理対象とする（前回失敗分のリトライ。`failure_reason` はStep 1.2で除去する）

**URL正規化ルール（SKILL.mdの「URL正規化ルール」セクション参照）：**
末尾スラッシュ除去、UTMパラメータ除去、www.除去、https統一。正規化は比較用のみ、frontmatterのsourceは元URLのまま。

### Step 1.2：frontmatter補完

frontmatterが欠損・不完全な場合、daily-summaryスキルの補完ルールに従い補完する。

- title: frontmatter.title → 本文の最初のH1 → ファイル名 → 本文冒頭1文（50字まで）
- source: frontmatter.source → 本文中の最初のURL → 空欄
- date: frontmatter.date → ファイル名中の日付パターン → 処理実行日
- tags: frontmatter.tags → `[]` で初期化

frontmatter自体がない場合は、ファイル先頭に `---` ブロックごと生成する。

前回失敗時の `failure_reason` がfrontmatterにある場合は、このステップで除去する。

### Step 1.5：各MDファイルにタグを付与する

各クリップMDファイルの本文を読んで内容を判断し、frontmatterの `tags:` にタグを追加する。
**タグは必ずStep 3でファイルを移動する前に書き込むこと。**
**タグ判定基準はdaily-summaryスキルのタグ体系セクションを必ず参照すること。**

**既存タグの扱い：**
- Web Clipperや手動で既に付いているタグは保持する
- daily-summary体系のタグ（`ジャンル/`, `重要度/`, `ソース/`）のみを追加する
- 同じカテゴリのタグが既にある場合は上書きしない（例：手動で `重要度/高` が付いていたらそれを尊重する）
- daily-summary体系以外のタグ（ユーザーが手動で付けた任意のタグ）には一切触れない

```yaml
tags:
  - my-custom-tag       # ← 手動タグ：保持する
  - ジャンル/AI          # ← 追加
  - 重要度/中            # ← 追加
  - ソース/SNS           # ← 追加
```

---

### Step 2：02_DailySummary/YYYYMMDD.md を生成する

YYYYMMDDは処理実行日。**ファイルが存在するかどうかで処理を分岐する。**

#### 追記時の重複チェック

パターンBで追記する場合、既存のDailySummaryファイル内に同じ `source:` URL（正規化後）の記事が既に掲載されていないか確認する。掲載済みの記事はスキップし、ログに記録する。

#### 字数制限（全体）
**1ファイルあたり1〜5万字に収めること。**
記事が多い日は重要度/低の要約をさらに短縮して調整する。

---

#### 重要度別の要約スタイル定義

各記事はStep 1.5で付与した `重要度/XX` タグに従って、以下のスタイルで要約する。

---

##### 🔴 重要度/高（10行程度・高密度）

業務への影響が大きい記事。表・比較・構造化を積極的に使って視覚性を高める。

```markdown
### N. 🔴 記事タイトル
> [記事タイトル](URL) — ソース名

#### 背景・文脈
（この記事が扱う問題・状況の背景を2〜3文で）

#### 要点
| # | 内容 |
|---|---|
| 1 | 要点1 |
| 2 | 要点2 |
| 3 | 要点3 |
| 4 | 要点4 |
| 5 | 要点5 |

#### 学んだこと・気づき
- 📌 **技術・知識として：** （何を知ったか）
- 💼 **業務への示唆：** （自分の仕事にどう活かせるか）
- 🔭 **次に調べること：** （この記事から派生する疑問・課題）

#### 重要な数字・固有名詞
`キーワード1` `キーワード2` `キーワード3`
```

---

##### 🟡 重要度/中（5〜7行程度・中密度）

近いうちに活用できそうな記事。要点を箇条書きで整理し、気づきを2観点で書く。

```markdown
### N. 🟡 記事タイトル
> [記事タイトル](URL) — ソース名

**要点**
- 要点1（記事の主張・結論）
- 要点2（具体的な数字・データ・固有名詞）
- 要点3（独自の視点や一般論との違い）

**学んだこと・気づき**
- 📌 **技術・知識として：** （何を知ったか）
- 💼 **業務への示唆：** （自分の仕事にどう活かせるか）
```

---

##### 🟢 重要度/低（3行程度・簡潔）

トレンド把握・参考程度の記事。1文の要旨と1行の気づきのみ。

```markdown
### N. 🟢 記事タイトル
> [記事タイトル](URL) — ソース名

**要旨：** （記事の内容を1〜2文で）
**気づき：** （1行で）
```

---

#### 復習クイズの品質基準

重要度に応じてクイズの質と数を変える。

| 重要度 | クイズ数 | 内容 |
|---|---|---|
| 高 | 3問 | 事実確認1問 ＋ 本質理解1問 ＋ 応用・業務活用1問 |
| 中 | 2問 | 事実確認1問 ＋ 本質理解1問 |
| 低 | 1問 | 事実確認1問のみ |

クイズの形式：
```markdown
（事実確認の問い）::（答え）
（本質理解の問い：「なぜ」「どういう意味で」を問う）::（答え）
（応用の問い：「自分の業務ならどう使う？」を問う）::（答え）
```

---

#### 複数記事がある場合：横断まとめセクションを追加

記事が2件以上ある日は、個別要約の後に以下を追加する。

```markdown
## 🔗 今日の共通テーマ

（今日の記事を横断して見えてくる共通のテーマ・文脈・トレンドを2〜3文で総括する。
 特定の記事の要約にならないよう注意。複数記事に共通するパターンや対比を中心に書く。）
```

---

#### パターンA：ファイルが存在しない場合（初回生成）

frontmatter＋本文をまるごと生成する。見出し構造はSKILL.mdの「DailySummaryの構造（固定見出し）」に従う。

```markdown
---
date: YYYY-MM-DD
tags:
  - flashcards
---

# 📅 Daily Summary: YYYY-MM-DD

## 今日クリップした記事（N件）

| # | タイトル | 重要度 | ジャンル |
|---|---|---|---|
| 1 | [記事タイトル](URL) | 🔴 高 | ジャンル/AI |
| 2 | [記事タイトル](URL) | 🟡 中 | ジャンル/テック |
| 3 | [記事タイトル](URL) | 🟢 低 | ジャンル/ツール |

---

## 記事ごとの要約・学び

（各記事を重要度別スタイルで出力）

---

## 🔗 今日の共通テーマ（記事が2件以上の場合のみ）

（横断まとめ）

---

## 今日のキーワード

`キーワード1` `キーワード2` `キーワード3`

---

## 復習クイズ

（各記事の重要度に応じた問数・質のクイズを出力）

---
*生成日時: YYYY-MM-DD HH:MM / 処理ファイル数: N件*
```

#### パターンB：ファイルが既に存在する場合（追記）

**frontmatterは書かない。** 既存ファイルの末尾に以下の形式で追記する。

```markdown

---

## ➕ 追記: HH:MM（N件）

| # | タイトル | 重要度 | ジャンル |
|---|---|---|---|
| 1 | [記事タイトル](URL) | 🟡 中 | ジャンル/AI |

---

## 記事ごとの要約・学び（追記分）

（各記事を重要度別スタイルで出力）

---

## 今日のキーワード（追記分）

`キーワード1` `キーワード2`

---

## 復習クイズ（追記分）

（重要度に応じたクイズを出力）

---
*追記日時: YYYY-MM-DD HH:MM / 追記ファイル数: N件*
```

### Step 2.5：処理済みマークを付与する

**Step 2（DailySummary生成）が成功した記事にのみ**、各MDファイルのfrontmatterに以下を追記する：

```yaml
processed: true
processed_at: "YYYY-MM-DDTHH:MM"
summary_date: "YYYYMMDD"
```

DailySummary生成に失敗した記事には `processed: true` を付与しない。これにより再実行時に自動的にリトライ対象になる。

### Step 3：00_Inboxのファイルを01_Clipsに移動する

- `01_Clips/YYYYMMDD/` フォルダを作成（存在する場合はスキップ）
- **処理成功したファイル（processed: true が付いたもの）のみ** `01_Clips/YYYYMMDD/` に移動
- **ファイル名衝突時：** 移動先に同名ファイルが既存の場合、SKILL.mdの「アーカイブ時のファイル名衝突ルール」に従う（同一URL→スキップ、異なるURL→リネーム）
- 失敗したファイルは 00_Inbox に残す
- 移動後に `00_Inbox` の残ファイルを確認する

### Step 3.5：失敗ファイルにfailure_reasonを記録する

処理に失敗して 00_Inbox に残ったファイルのfrontmatterに、失敗理由を記録する：

```yaml
failure_reason: "読み込みエラー: frontmatter解析失敗"
failure_at: "YYYY-MM-DDTHH:MM"
```

これにより、00_Inbox内で未処理ファイルと失敗ファイルを区別できる。次回実行時にはStep 1.1で再処理対象として扱い、Step 1.2でfailure_reasonを除去してリトライする。

### Step 4：99_Logs/YYYYMMDD.md にログを記録する

ログは既存ファイルがあれば追記、なければ新規作成する。

```markdown
## YYYY-MM-DD HH:MM 処理ログ

- **コマンド**: /daily-summary
- **処理対象ファイル数**: N件
- **成功**: N件
- **スキップ（既処理）**: N件
- **スキップ（重複）**: N件
- **失敗**: N件
- **生成ファイル**: 02_DailySummary/YYYYMMDD.md（新規 or 追記）
- **アーカイブ先**: 01_Clips/YYYYMMDD/

### ファイル別処理結果

| ファイル名 | 結果 | 備考 |
|---|---|---|
| article1.md | ✅ 成功 | 重要度/高 |
| article2.md | ✅ 成功 | 重要度/中 |
| article3.md | ⏭ スキップ | 既処理（processed: true） |
| article4.md | ⏭ スキップ | 重複（source URL一致） |
| article5.md | ❌ 失敗 | 読み込みエラー → failure_reason記録済み |
```

---

## 完了後の報告

```
✅ daily-summary 完了
- 処理した記事: N件（成功N / スキップN / 失敗N）
- まとめ: 02_DailySummary/YYYYMMDD.md
- アーカイブ: 01_Clips/YYYYMMDD/
- ログ: 99_Logs/YYYYMMDD.md
```

失敗やスキップがある場合は理由を簡潔に報告する。

## エラー処理

| 状況 | 対応 |
|---|---|
| ファイル読み込み失敗 | スキップして他を処理、ログに記録、ファイルに failure_reason を記録して 00_Inbox に残す |
| frontmatter欠損 | 補完ルールに従い補完してから処理を続行 |
| DailySummaryが既に存在 | 末尾に追記（上書きしない）。追記前にsource URL重複チェック（正規化済み） |
| 01_Clips移動失敗 | まとめは保存済み、エラーをログに記録しユーザーに通知 |
| 01_Clipsファイル名衝突 | 同一URL→スキップ、異なるURL→リネーム |
| 全件失敗 | DailySummaryは生成しない、ログに全件の失敗理由を記録 |

## 再実行時の動作

再実行しても安全に動作する。具体的には：

- `processed: true` のファイルはStep 1.1で自動除外される
- `failure_reason` 付きのファイルは再処理対象としてリトライされる
- DailySummaryへの追記時はsource URL（正規化済み）で重複チェックされる
- 01_Clipsに既に存在するファイルは移動をスキップする
- ログは追記モードで既存ログを上書きしない
